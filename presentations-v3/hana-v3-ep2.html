② 背後的技術 · 為什麼不用 MCP

MCP 這條路，會逼你掏出 API Key

用 MCP 連 Claude
MCP 本身有其限制；
Claude 不支援非官方工具走 OAuth；
→ 一定得掏出一組 API Key。
直接開官方 CLI
不用碰 MCP 的限制；
複用你已經登入的訂閱帳號；
→ 不必額外花錢買 API 額度。
這不是我隨口的選擇，是為了守住「只靠訂閱帳號」這條初衷，不得不繞開 MCP。
② 背後的技術 · 三種取捨

訂閱橋接 vs API Key vs 本地模型

方式 成本 合法性 能力
訂閱橋接（我走的路）就是原本的訂閱費，不必額外付錢官方 CLI 的正常使用方式官方模型完整能力
API Key隨用量飆升，另一筆額外開銷沒問題官方模型完整能力
本地模型前期硬體成本高沒問題通常比不上雲端大模型
訂閱橋接，是唯一同時滿足「便宜、合法、能力完整」的一條路。
③ 👉 你可以這樣自己做

三步驟，你也能搭一個引擎

1
裝好一個官方 CLI，登入你自己的訂閱帳號。
2
用程式開一個子行程執行它，把提示詞從標準輸入餵進去。
3
讀它的標準輸出，把回覆接回來顯示——這就是引擎。
不必自己接 API，也不必養模型——你已經登入的 CLI，就是現成的大腦。
④ 誠實邊界

CLI 橋接的代價

受限於 CLI 介面
CLI 有什麼功能，我才能用什麼功能——不是我自己定義的能力邊界。
得遵守它的限流規則
訂閱帳號本身的用量限制，就是我的限制。
不是官方穩定的 API 契約
我依賴的是命令列介面的行為，介面改版，我就得跟著調整。
這是我為了不必掏 API Key，願意承擔的取捨——誠實講給你聽，不誇大也不隱瞞。
